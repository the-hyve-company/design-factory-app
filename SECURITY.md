# Security Policy

## Supported versions

Design Factory is pre-1.0 and ships from `main`. Security fixes land in the
next release; we do not backport to tagged 0.x versions.

| Version         | Supported   |
| --------------- | ----------- |
| `main` (latest) | yes         |
| `0.1.x`         | best effort |
| < `0.1`         | no          |

## Reporting a vulnerability

Please **do not** open a public issue for security vulnerabilities.

Send a private report to the maintainers via GitHub Security Advisories:

1. Go to the repository's [Security tab][advisories].
2. Click "Report a vulnerability".
3. Describe the issue with reproduction steps, affected versions and impact.

We aim to:

- **Acknowledge** within 3 business days.
- **Triage and confirm** within 7 business days.
- **Patch** within 30 days for critical issues, 90 days for lower severity.

## Threat model

Design Factory is a single-user app with a localhost daemon. The trust
boundary is the machine running the repo: anyone with shell access
already controls everything the daemon does. The relevant adversary is
an HTML payload generated by the LLM (or pasted in by the user) that
tries to escalate from the iframe → the daemon → the filesystem.

### Iframe sandbox model

The preview iframe runs project HTML. The daemon never injects scripts
into the iframe; the model does.

Two sandbox modes:

- **Permissive** (`sandbox="allow-scripts allow-same-origin"`) — the
  default. Required by four DOM-coupled features in the current
  surface: inline text edit, comment-mode click handler, in-place
  patch DOM mutation, and the animated-scene transport bridge. Each of
  these uses `iframe.contentDocument` directly; strict sandbox would
  break them.

- **Strict** (`sandbox="allow-scripts"` only, no `allow-same-origin`) —
  opt-in via URL param `?strictSandbox=1` or localStorage key
  `DF_STRICT_SANDBOX=1`. Disables the four features above.

The tweaks-bridge and element-overlay paths are postMessage-only and
forward-compatible with strict sandbox. Migrating the remaining
DOM-coupled features to postMessage would let strict sandbox become the
default without breaking surface area.

The default is permissive because flipping it silently would break
four user-visible features in the current build. Users who don't need
those features can opt into strict mode today via the flag above.

### Daemon filesystem boundary

The daemon listens on `127.0.0.1` and rejects browser requests whose
`Origin` is not in the local allowlist. The terminal WebSocket applies
the same origin check during the upgrade handshake.

Generic filesystem endpoints are scoped by default to Design Factory
workspace roots: `projects/`, `design-systems/`, `skills/`, plus
read-only access to the bundled `landing/` preview. Writes outside
those roots are rejected with `PATH_OUT_OF_SCOPE`. Power users can opt
into unrestricted local filesystem access for private workflows by
starting the daemon with:

```bash
DF_ALLOW_ARBITRARY_FS=1 npm run dev:web
```

That flag is intentionally off by default and should not be used while
testing untrusted generated HTML.

The `/fs/write/artifact` endpoint is stricter: it always scopes writes
under `projects/<slug>/`, uses realpath-based containment via
`assertPathInScope`, and validates content via
`validateArtifactStaticP0Minimal` (byte floor + HTML-shape sanity)
before any disk write.

## Scope

In scope:

- Code execution via crafted prompts, project HTML, design system imports
  or skill scripts that escapes the iframe sandbox or the daemon's
  filesystem boundary.
- Bypass of the daemon's `/fs/*` path containment.
- Token leaks through error messages, logs or shared bundles (Share menu,
  exported HTML).
- Iframe → parent escape via `allow-same-origin` permissive mode IF the
  payload is generated by Design Factory's own UI without user consent.
  (User-pasted/imported HTML is the user's responsibility.)

Out of scope (but still nice to know about):

- Bugs in the upstream provider CLIs (Claude, Codex, Gemini, Ollama).
- Issues that require a malicious user already having shell access on
  the same machine.
- DoS by overloading the local daemon (single-user app).
- Strict-sandbox feature breakage when opt-in flag is set — those
  features are documented as DOM-coupled and incompatible with
  `allow-scripts`-only mode.
